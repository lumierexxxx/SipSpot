# 第二章 需求分析

本章从用户角色、功能需求、非功能需求和核心用例四个维度对 SipSpot 平台进行系统性分析。需求分析是系统设计与实现的前提，通过明确各类用户的使用场景与操作权限、梳理平台的核心功能边界以及量化非功能性质量指标，为后续的系统架构设计提供依据。

---

## 2.1 用户角色分析

SipSpot 平台面向三类用户群体，不同角色拥有不同的权限范围与操作能力。根据访问权限与业务职责的差异，将用户划分为游客、注册用户和店主/管理员，具体如表2-1所示。

**表2-1 用户角色与权限**

| 用户角色 | 描述 | 主要操作 |
|---------|------|---------|
| 游客（未登录） | 未注册或未登录的访客 | 浏览咖啡馆列表、查看详情、使用语义搜索、查看地图 |
| 注册用户 | 已注册并登录的普通用户 | 上述所有操作 + 撰写评论、收藏咖啡馆、获取个性化推荐、管理个人资料 |
| 店主/管理员 | 拥有店主权限的用户 | 上述所有操作 + 发布咖啡馆、编辑/删除自己的咖啡馆、管理咖啡馆图片 |

**游客**是平台的轻量访问者，无需注册即可享有完整的信息浏览与搜索功能，包括使用自然语言语义搜索和高德地图附近查询。这一设计降低了用户的进入门槛，有助于提升平台的曝光率与内容传播效果。

**注册用户**是平台的核心参与群体。在游客功能基础上，注册用户可撰写多维度评分评论、将心仪咖啡馆加入收藏，并基于自身历史行为获取 AI 个性化推荐。用户的每一次评论和收藏行为都会更新其偏好向量，从而持续优化推荐质量。

**店主/管理员**是平台的内容生产者。在注册用户权限之上，店主可创建和维护咖啡馆信息页面，包括上传图片、管理营业时间与设施信息，以及编辑或删除归属于自身账号的咖啡馆条目。系统通过后端中间件 `checkOwnership` 确保店主仅能操作自己创建的内容，防止越权修改。

## 2.2 功能需求

### 2.2.1 核心 AI 功能需求

本平台的差异化竞争力在于将 AI 能力深度融入咖啡馆发现流程，主要体现在语义搜索、评论情感分析和个性化推荐三个方面。

**1. 自然语言语义搜索**

传统关键词搜索要求用户精确匹配预定义标签，难以表达模糊的情景化需求。本平台允许用户以自然语言描述搜索意图，例如"安静适合写代码的地方"或"有氛围感的下午茶空间"。系统使用 `multilingual-e5-base` 多语言语义模型将查询文本编码为768维稠密向量，在应用层对数据库中所有已生成 embedding 的咖啡馆逐一计算余弦相似度，按相似度降序返回 Top-K 结果。该方案支持中英文混合查询，能够识别语义近似但表述不同的需求（如"办公"与"适合工作"），显著提升搜索召回的语义准确性。

**2. 评论情感分析**

用户提交评论后，系统异步调用通义千问（Qwen）API 对评论文本进行情感分析，输出结果包括：情感倾向（positive / negative / neutral）、3至5个关键词、一句话摘要以及置信度评分。分析结果写回评论文档，并汇聚至咖啡馆详情页的 AI 综合摘要区域，帮助后续用户快速了解该咖啡馆的整体口碑。考虑到 API 网络可用性，系统设有本地降级机制：当通义千问 API 不可访问时，自动切换至基于本地关键词词典的规则匹配方法 `basicSentimentAnalysis()`，确保情感分析功能的持续可用性。

**3. 个性化推荐**

系统通过对用户的评论历史和收藏行为进行加权聚合，生成用户级别的768维偏好向量，持久化存储于用户文档的 `preferenceEmbedding` 字段。推荐引擎采用双路径策略：当偏好向量存在时，走向量路径，使用余弦相似度将偏好向量与咖啡馆 embedding 进行匹配，返回语义最相关的咖啡馆；当偏好向量尚未建立（新用户或行为数据不足时），走规则路径，依据 `specialty`、`amenity`、`vibe` 等结构化字段进行过滤排序。此外，系统还提供"探索推荐"功能，主动推送用户尚未尝试过的咖啡种类，鼓励探索多样化的消费体验。

### 2.2.2 基础平台功能需求

除 AI 核心能力外，平台还需提供完整的信息管理与社区交互基础功能。

**1. 咖啡馆信息管理**

支持咖啡馆条目的完整 CRUD 操作。店主可填写名称、地址、设施（amenities）、专长咖啡种类（specialty）、营业时间等结构化信息，并上传多张图片。图片经由 `multer` 处理后上传至 Cloudinary 云存储，返回持久化 URL 存入数据库，避免本地存储带来的运维负担。

**2. 多维度评分系统**

用户评论支持总体评分与五个细分维度评分（口味、价格/性价比、环境、服务态度、办公适合度），评分步长为0.5星，范围为1至5星。细分评分帮助用户在不同使用场景下（如商务办公 vs. 休闲聚会）获取更有针对性的参考信息，同时为 AI 个性化推荐提供更丰富的偏好信号。

**3. 地理位置功能**

咖啡馆地理坐标以 GeoJSON Point 格式存储，支持 MongoDB 的 `$near` 地理查询，实现"附近咖啡馆"功能。前端集成高德地图（AMap）SDK，在地图视图中展示周边咖啡馆的点位标注，用户可切换列表视图与地图视图，直观感受空间分布。

**4. 用户账户管理**

提供完整的账户生命周期管理，包括注册（用户名/邮箱/密码）、登录、密码修改及邮箱验证。认证基于 JWT，令牌存储于 httpOnly Cookie，刷新令牌机制保障会话连续性。用户可在个人资料页更新昵称、头像及偏好设置。

**5. 收藏功能**

注册用户可对任意咖啡馆执行收藏/取消收藏操作，收藏状态实时同步至个人收藏列表页面。收藏行为同时作为偏好信号参与用户偏好向量的更新计算，形成"行为 → 偏好向量 → 推荐优化"的正向反馈闭环。

## 2.3 非功能需求

平台的非功能需求涵盖性能、安全、可用性、可扩展性和可维护性五个维度，具体指标如表2-2所示。

**表2-2 非功能需求**

| 类别 | 需求描述 | 指标 |
|------|---------|------|
| 性能 | API 响应时间（不含 AI 推理） | < 500ms |
| 性能 | Embedding 推理耗时 | < 3s（首次加载模型除外） |
| 安全 | 认证方式 | JWT 存储于 httpOnly Cookie，防 XSS 攻击 |
| 安全 | 输入校验 | Joi Schema 校验所有入参 |
| 安全 | 请求限流 | 认证接口 5 次/分钟；通用接口 100 次/15 分钟 |
| 可用性 | AI 模块降级 | 向量模型未就绪时回退关键字搜索；Qwen 不可用时回退本地情感分析 |
| 可扩展性 | 向量存储 | 当前在 MongoDB 存储 embedding，设计上支持迁移至专用向量数据库 |
| 可维护性 | 代码规范 | TypeScript 类型安全，ESLint 校验，JSDoc 注释 |

在性能层面，所有非 AI 推理接口的响应时间应控制在500毫秒以内，确保用户在浏览和交互时不产生明显延迟。在安全层面，系统采用 httpOnly Cookie 存储 JWT，有效抵御跨站脚本攻击（XSS），并通过 `express-rate-limit` 对高风险的认证接口实施严格限流，防止暴力破解。在可用性层面，AI 模块的所有关键路径均设有降级策略，确保即便外部 API 服务中断，核心浏览与搜索功能仍能正常运行。

## 2.4 用例描述

本节选取三个代表性核心用例进行详细描述，涵盖 AI 语义搜索、情感分析触发和个性化推荐三条最具业务价值的功能路径。

---

**用例编号**：UC-01
**用例名称**：自然语言语义搜索咖啡馆
**参与者**：游客 / 注册用户
**前置条件**：用户打开搜索页面，Embedding 模型已完成初始化加载
**主要流程**：
1. 用户在搜索框输入自然语言查询（如"安静适合写代码的地方"）
2. 系统将查询文本提交至 `/api/cafes/ai-search` 接口
3. 后端调用 `generateEmbedding(query, 'query')` 生成768维查询向量
4. `vectorService.rankCafes()` 对所有已有 embedding 的咖啡馆计算余弦相似度
5. 按相似度降序排列，返回 Top-K 结果列表
6. 前端渲染结果列表，每条结果附带语义相关度分数

**替代流程**：若 Embedding 模型未就绪（`isReady()` 返回 `false`），系统回退至基于关键词正则匹配的传统搜索，返回包含关键词的咖啡馆列表，并在响应中标注降级状态。

**后置条件**：用户看到与查询语义相关的咖啡馆列表，结果按相关度由高到低排列。

---

**用例编号**：UC-02
**用例名称**：提交评论触发情感分析
**参与者**：注册用户
**前置条件**：用户已登录，且正在查看某咖啡馆详情页
**主要流程**：
1. 用户填写评论内容和各维度评分，点击"提交评论"按钮
2. 前端调用 `cafesAPI.createReview()` 将评论数据发送至后端
3. 系统将评论文档写入数据库，并向前端返回成功响应
4. 后台异步调用 `analyzeReview()` 触发通义千问 API 情感分析
5. 分析结果（`sentiment` / `keywords` / `summary` / `confidence`）写回评论文档的 `aiAnalysis` 字段
6. 咖啡馆详情页的 AI 综合摘要区域在下次加载时展示更新后的情感分析内容

**替代流程**：若 Qwen API 不可用，系统自动调用本地 `basicSentimentAnalysis()` 进行降级分析，基于预定义关键词词典输出情感倾向，确保 `aiAnalysis` 字段不为空。

**后置条件**：评论已成功保存至数据库，情感分析结果写入评论文档并在详情页可见。

---

**用例编号**：UC-03
**用例名称**：获取个性化推荐
**参与者**：注册用户
**前置条件**：用户已登录，且至少有过一次评论或收藏行为以建立偏好数据
**主要流程**：
1. 用户访问推荐页面，前端发起 `GET /api/recommendations/me/personalized` 请求
2. 系统读取当前用户文档中的 `preferenceEmbedding`（768维偏好向量）
3. 若偏好向量存在，调用 `vectorService.rankCafes()` 计算偏好向量与所有咖啡馆 embedding 的余弦相似度，返回语义最相近的咖啡馆列表
4. 若偏好向量尚未建立，系统切换至规则路径，依据用户历史偏好的 `specialty`、`amenity`、`vibe` 字段进行过滤与排序
5. 同时发起 `GET /api/recommendations/me/explore` 请求，推送用户尚未尝试过的咖啡种类对应的咖啡馆（探索推荐）
6. 前端将个性化推荐列表与探索推荐列表分区展示

**后置条件**：用户看到基于自身行为偏好生成的个性化推荐列表，以及拓展消费多样性的探索推荐列表。
