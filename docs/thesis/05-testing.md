## 5.3 AI 搜索质量评估

### 5.3.1 对比实验设计

为验证语义搜索相较于传统关键字搜索的实际效果，本节设计对比实验，在含 50 条测试数据的本地数据库中，分别使用语义搜索（基于 multilingual-e5-base 向量模型）与关键字搜索（MongoDB `$text` 全文检索）对相同查询文本进行检索，对比返回结果的相关性。

**表5-3 语义搜索 vs 关键字搜索对比实验**

| 查询文本 | 语义搜索结果（Top-3 相关特征） | 关键字搜索结果 | 语义优势说明 |
|---------|-------------------------------|-------------|------------|
| "安静适合写代码的地方" | 含 WiFi、安静环境、适合工作 / 办公属性的咖啡馆 | 仅匹配描述中含"代码"或"安静"字面的记录 | 理解"写代码"背后的办公需求场景 |
| "有氛围感的下午茶" | 含 vibe=cozy / relaxed 氛围标签的咖啡馆 | "下午茶"无精确字面匹配，返回空或不相关结果 | 理解"氛围感"概念，跨越字面限制 |
| "cheap coffee near work" | 价格实惠、评分高、办公友好的咖啡馆 | 英文关键字无法匹配中文数据库 | 支持跨语言语义检索 |
| "人少不吵闹" | 含"安静环境"属性的咖啡馆 | 无"人少"或"不吵闹"的精确字段匹配 | 同义表达语义等价识别 |
| "适合拍照打卡" | 含 vibe=instagram-worthy 标签的咖啡馆 | 无相关字面匹配 | 概念语义理解与标签映射 |

### 5.3.2 实验结果分析

以上对比实验结果表明，语义搜索在模糊语义查询场景下表现明显优于关键字搜索。

系统将查询文本与咖啡馆文档（包含名称、描述、amenities 标签、vibe 标签等字段）统一映射至 multilingual-e5-base 模型的 768 维向量空间，通过余弦相似度进行排序与检索。实验数据显示，语义相关结果的余弦相似度普遍高于 0.75，而关键字搜索在处理同义词、多语言查询和概念性描述时存在明显局限——"不吵闹"与"安静环境"在字面层面无法匹配，英文查询在纯中文数据库中几乎无法返回有效结果。

multilingual-e5-base 模型对中英文混合输入的支持，使系统具备跨语言语义搜索能力，这对面向国际化用户或混合语言使用场景的咖啡馆发现应用具有实际价值。

## 5.4 性能测试

### 5.4.1 测试方法

性能测试采用 Postman 手动发送请求并记录响应时间的方式进行抽样测试，每个接口在预热完成后连续测试10次，取平均值与最大值作为参考指标。测试数据库包含约 50 条咖啡馆记录，模拟轻量级生产环境数据规模。

**表5-4 API 接口性能测试结果**

| 接口 | 平均响应时间 | 最大响应时间 | 说明 |
|------|------------|------------|------|
| `POST /api/auth/login` | 180ms | 320ms | 含 bcrypt 密码验证（10 轮哈希）|
| `GET /api/cafes` | 45ms | 120ms | 分页查询，20 条/页，含索引 |
| `POST /api/cafes/ai-search` | 850ms | 1400ms | 含 Embedding 推理 + 应用层向量排序 |
| `GET /api/recommendations` | 620ms | 980ms | 含向量相似度排序（50 条候选记录）|
| `POST /api/reviews/:id/analyze` | 1200ms | 2100ms | 含 Qwen API 远程调用（受网络延迟影响）|

### 5.4.2 性能分析

从测试结果来看，常规数据库查询接口（如分页列表、用户信息等）响应时间均在 50ms 以下，性能表现良好。含 bcrypt 验证的登录接口平均耗时 180ms，主要开销来自 10 轮哈希计算，属于安全性与性能之间的合理权衡。

AI 相关接口的响应时间相对较高。语义搜索接口（`/ai-search`）在服务启动后首次请求需加载约 280MB 的模型文件，耗时可达数秒；模型预热完成后，每次推理耗时约 100–150ms，应用层对 50 条记录进行余弦排序的附加耗时约 10ms，总体响应时间基本可接受。Qwen API 情感分析接口耗时最高，平均 1200ms，主要受网络 I/O 和远程模型推理延迟影响，波动较大。

针对上述性能瓶颈，建议后续对情感分析任务采用异步处理策略：评论提交后立即返回成功响应，AI 分析在后台异步执行并更新数据库，避免阻塞用户操作，从而改善前端感知响应速度。

## 5.5 安全测试

安全测试针对系统已实施的三项核心安全机制进行验证，确认各防护措施在实际请求场景下的有效性。

**（1）NoSQL 注入防护**

测试方法：在登录接口的 `password` 字段中传入 MongoDB 操作符 `{ "$gt": "" }`，尝试绕过密码验证。

测试结果：`express-mongo-sanitize` 中间件在请求到达控制器之前对请求体进行清洗，将 `$` 前缀的键名替换为无害字符，注入操作符失效，接口正常返回登录失败（400），数据库查询未受影响。防护机制有效。

**（2）速率限制验证**

测试方法：在同一分钟内对 `POST /api/auth/login` 接口连续发送 6 次请求（无论成功与否）。

测试结果：前 5 次请求正常处理，第 6 次请求收到 `429 Too Many Requests` 响应，响应体包含重试时间提示。此行为与 `express-rate-limit` 配置（认证接口严格限速：5 次/分钟）完全吻合，有效防止暴力破解攻击。

**（3）JWT 过期与自动续期**

测试方法：使用已过期的 access token 发送 `GET /api/users/me` 请求，观察前端行为。

测试结果：服务端返回 `401 Unauthorized`，前端 Axios 拦截器捕获该错误后，自动调用 `POST /api/auth/refresh` 接口使用 refresh token 获取新的 access token，并将原始请求重新发送，整个过程对用户完全透明，无感知续期机制正常工作。

综合安全测试结果，系统在注入防护、访问频率控制和令牌生命周期管理三个维度的安全机制均运行正常，符合设计预期。
